# Benchmark Freeze Statement

## Freeze scope

The benchmark freeze is **benchmark-only**. The frozen benchmark includes:

- scenario set
- enforcement contracts
- oracle specification
- evaluator logic
- diagnosis logic
- metrics definitions
- runtime F1 exclusions
- methodological assumptions

The freeze does **not** include model execution profiles. Model profiles are treated as execution conditions applied to the frozen benchmark and may be added or changed later, provided the frozen benchmark artifacts remain unchanged and the new profiles are documented in the execution manifest.

## Freeze readiness criteria

The benchmark may be frozen only if all of the following are true:

- all scenarios have complete schema and coherent outcome semantics
- no orphan runtime rules exist
- no contract-declared rules are unused by the runtime
- all runtime F1 exclusions are explicitly justified
- `evaluate.py` and `diagnose_f1.py` produce consistent diagnostics
- the pre-freeze audit contains no rows in `needs_fix`
- the calibration log documents every change applied during the pre-freeze stage
- the pre-freeze validation executes without errors

## Materialize review rule

`python3 -m tools.enforcement.materialize --out .` is treated as deterministic regeneration, not as an automatic source of truth.

Every materialization step must be followed by explicit review of:

```bash
git diff -- benchmark/enforcement/scenarios contracts/enforcement
```

No autogenerated change may be accepted into the freeze unless it has been manually reviewed and recorded in the pre-freeze audit or calibration log.

## Frozen artifact identity

The frozen benchmark identity is defined by:

- `benchmark/enforcement/benchmark_manifest.json`
- the freeze commit recorded in that manifest
- the benchmark version recorded in that manifest

The benchmark manifest must include hashes for the key frozen artifacts:

- scenarios
- contracts
- oracle specification
- evaluator
- diagnosis

## Post-freeze versioning policy

| Change after freeze | Required action |
| --- | --- |
| Scenario changes | Create `benchmark-v1.1` and rerun the full experiment |
| Contract changes | Create `benchmark-v1.1` and rerun the full experiment |
| Oracle changes | Create `benchmark-v1.1` and rerun the full experiment |
| Evaluator changes | Create `benchmark-v1.0.1` if it is a bugfix, but rerun the full experiment |
| Add a model | Do not change the benchmark version; register the model in the execution manifest |
| Change a model profile | Do not change the benchmark version; register the new condition in the execution manifest |

## Methodological assumptions

- The tools are deterministic mocks.
- The benchmark is a controlled evaluation artifact, not a production deployment study.
- The primary claims concern runtime governance behavior under controlled conditions.
- External validity to multi-agent settings, production APIs, or adversarial network conditions is outside the benchmark freeze scope unless added in a later benchmark version.

## Current freeze state

This statement is not yet declaring the benchmark frozen. It defines the criteria and policy that must be satisfied before the freeze can be formally declared.
